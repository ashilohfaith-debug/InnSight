System Architecture & Database Schemas

Detailed technical specifications of database indexing structures, bitwise scheduling grids, and asynchronous worker sequence diagrams.

Database Schema: listings

A listings row uses PostGIS coordinates spatial geometry formatting for bounding index searches, and daily calendars are stored as bitmap bitmask vectors to prevent fractional checkout collision.

PostgreSQL + PostGIS listings schema
CREATE TABLE listings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  address VARCHAR(512) NOT NULL,
  -- PostGIS geometry type: 2D Point, WGS84 spatial reference
  location GEOMETRY(Point, 4326) NOT NULL,
  base_rate NUMERIC(10, 2) NOT NULL,
  cleaning_fee NUMERIC(10, 2) NOT NULL,
  tax_rates JSONB NOT NULL, -- Regional tax coefficients array
  -- 24h Availability Bitmap Array Segment (24 bits)
  availability_grid BIT(24) DEFAULT X'FF0F00',
  unverified_amenities VARCHAR(255)[],
  ground_truth_summary JSONB DEFAULT '{}'
);

-- R-Tree Indexing for high-speed spatial point queries
CREATE INDEX listings_location_gix ON listings USING GIST (location);
Fractional Late Checkout bitwise AND check
-- Input: Requested late checkout extension hours 11:00 to 13:00 (2h)
-- Requested mask:   000000000001100000000000 (0x018000)
-- Cleaner/next reservation block mask: 
--                   000000000000011000000000 (0x006000)

CREATE OR REPLACE FUNCTION verify_extension(
  listing_id UUID, 
  requested_mask BIT(24)
) RETURNS BOOLEAN AS $$
DECLARE
  blocked_slots BIT(24);
BEGIN
  SELECT (availability_grid # X'FFFFFFFF') INTO blocked_slots 
  FROM listings WHERE id = listing_id;
  
  -- Bitwise AND: checks if requested slots overlap with blocked blocks
  IF (requested_mask & blocked_slots) = X'000000' THEN
    RETURN TRUE; -- Free to extend
  ELSE
    RETURN FALSE; -- Conflict exists
  END IF;
END;
$$ LANGUAGE plpgsql;

Asynchronous AI Audit Pipelines

When property assets are uploaded, the client receives `HTTP 202 Accepted` to prevent UI thread blocking. Media files are stored in AWS S3, and tracking tasks are queued in Redis for Celery worker processing.

Zero-Shot Computer Vision Tagging (CLIP Model) & EXIF GPS Audit

The zero-shot computer vision worker utilizes a Hugging Face CLIP model to calculate the mathematical Cosine Similarity between the uploaded image vector \(\vec{v}\) and the target amenity text description vector \(\vec{u}\):

\[\text{Similarity} = \frac{\vec{v} \cdot \vec{u}}{\|\vec{v}\| \|\vec{u}\|}\]

Verification Rules:
1. If similarity \(< 0.85\) for any amenity claimed by the host (e.g. "dedicated workspace"), the listing is automatically flagged with `unverified_amenities = ['Dedicated Workspace']`.
2. The EXIF parser extracts embedded coordinate points and cross-references them with the database PostGIS address coordinate geometry point: if distance \(> 100\text{ meters}\), listing verification fails.
